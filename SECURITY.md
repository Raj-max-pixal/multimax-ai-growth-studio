# 🚨 Reporting a Vulnerability

If you discover a security vulnerability, **please do not create a public GitHub issue**.

Instead:

1. Gather as much information as possible.
2. Describe how the issue can be reproduced.
3. Explain the potential impact.
4. Contact the project maintainers privately.

Please include:

* Vulnerability description
* Steps to reproduce
* Expected behavior
* Actual behavior
* Screenshots (if applicable)
* Environment details
* Suggested fix (optional)

---

# ⏱️ Response Timeline

| Stage                      | Estimated Time                 |
| -------------------------- | ------------------------------ |
| Initial Acknowledgement    | Within 48 Hours                |
| Investigation              | 3–7 Days                       |
| Resolution (when possible) | Based on Severity              |
| Public Disclosure          | After the issue has been fixed |

---

# 🔐 Security Best Practices

We encourage everyone to:

* Keep dependencies up to date.
* Never commit secrets or API keys.
* Use environment variables for sensitive information.
* Enable Two-Factor Authentication (2FA) on GitHub.
* Follow secure coding practices.
* Report suspicious behavior responsibly.

---

# 🚫 Out of Scope

The following are generally **not considered security vulnerabilities**:

* Spelling or grammar mistakes
* UI or design suggestions
* Feature requests
* Performance improvements
* Duplicate reports
* Issues caused by unsupported software versions

---

# 🤝 Responsible Disclosure

We ask that you:

* Give maintainers reasonable time to investigate and fix the issue.
* Avoid publicly disclosing vulnerabilities before a fix is available.
* Respect user privacy and data.
* Act in good faith throughout the disclosure process.
